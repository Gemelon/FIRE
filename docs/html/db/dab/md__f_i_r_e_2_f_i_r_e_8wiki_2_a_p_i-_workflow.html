Pipeline Model

The API is organized as a deterministic three-step workflow:

  1. Collect
  2. Generate
  3. Execute

Each step has a dedicated concern and can be run independently for validation.

1) Collect

Purpose:

  • scan configured source directories
  • resolve metadata from configured providers
  • persist file records into SQLite

Typical outcomes:

  • metadata normalization
  • sidecar detection and classification
  • complete source inventory snapshot

2) Generate

Purpose:

  • evaluate SortingPatern and FileNamePatern
  • resolve placeholders from metadata
  • compute final target paths

Typical outcomes:

  • deterministic output paths
  • conflict visibility before file operations
  • persisted generation state for review

3) Execute

Purpose:

  • apply configured operation (Copy, Move, Link)
  • process regular and sidecar files
  • report operation progress and results

Typical outcomes:

  • physical media reorganization
  • traceable final operation results

Inspection and Diagnostics

The API also supports targeted metadata inspection via helper methods and console commands. These capabilities are useful for configuration tuning and troubleshooting.

Host Application Guidance

Host applications should:

  • call API steps in sequence
  • subscribe to progress/state events
  • treat console output as optional adapter behavior
