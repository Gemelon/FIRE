void CollectFiles ( Action< string >? progressCallback = null)

Step 1: Collects all files and their metadata from configured root directories.

Parameters
progressCallbackOptional callback invoked for each file being processed. The callback receives the file path. Use this to report progress in a UI or log file.

This method:

  1. Clears any existing records from the database.
  2. Recursively scans all paths in FIREConfigration.FilesRootPath.
  3. For each file matching a configured file extension:
    • Retrieves file ID information from the OS.
    • Queries metadata from all configured sources and keywords.
    • Creates a FIREDbRecord and adds it to the database.
  4. Saves all records to the database when complete.

Errors during file access or metadata extraction are logged but do not stop the pipeline.

Orchestrates the three-step FIRE processing pipeline: collect, generate, and execute.

FIRECatalog is the main orchestrator that drives metadata collection from files, generates target directory and file names using path templates, and executes file operations (Copy, Move, or Link). It integrates the configuration, database, and metadata sources to implement the complete file processing workflow.

Three-Step Workflow:

  1. Collect: CollectFiles recursively scans all configured root directories, creates file records for each file, and populates metadata from configured sources.
  2. Generate: GenerateTargetPaths computes output directory and file name for each record by parsing path templates with resolved metadata values.
  3. Execute: ExecuteFileOperations performs the configured file operation (Copy, Move, or Link) from source to target path.

Windows File Identification: FIRECatalog uses Windows API calls (GetFileInformationByHandleEx) to retrieve volume serial numbers and file IDs, which enable detection of hard links and files on the same physical volume. This is done on Windows/NTFS systems only.

Extensibility: The metadata extraction system is extensible via MetadataSourceRegistry. Custom metadata sources can be registered to support additional file formats or sources.

Resource Management: This class implements IDisposable to properly clean up the database and metadata sources. Always dispose of this instance when done.

Using FIRECatalog for the complete pipeline:

var config = FIREConfigration.Load("Configuration.yaml");
config.EnsureSupportedConfigurationVersion();
var dbPath = Path.Combine(config.DataBasePath, config.DataBaseFileName);
using var database = new FIREDatabase(dbPath);
using (var catalog = new FIRECatalog(config, database))
{
// Step 1: Collect metadata
catalog.CollectFiles(path => Console.WriteLine($"Collecting: {path}"));
// Step 2: Generate target paths
catalog.GenerateTargetPaths(path => Console.WriteLine($"Generating: {path}"));
// Step 3: Execute file operations
catalog.ExecuteFileOperations(path => Console.WriteLine($"Processing: {path}"));
}

void ExecuteFileOperations ( Action< string >? progressCallback = null)

Step 3: Executes the configured file operation (Copy, Move, or Link) for all records.

Parameters
progressCallbackOptional callback invoked for each file being processed. The callback receives the file path.

This method iterates over all records and performs the configured file operation:

  • Skips records with null or invalid paths or files that no longer exist.
  • Looks up the file extension configuration.
  • Retrieves the action (Copy, Move, or Link) using extension-specific overrides if available.
  • Calls ExecuteAction to perform the operation.

The operation creates all necessary output directories and handles errors gracefully. All changes are persisted to the database when complete.

ObjectDisposedExceptionThrown if this instance has been disposed.

◆ GenerateTargetPaths()

void GenerateTargetPaths ( Action< string >? progressCallback = null)

Step 2: Generates target file paths for all collected file records.

Parameters
progressCallbackOptional callback invoked for each file being processed. The callback receives the file path.

This method iterates over all records in the database and computes the target directory and file name by:

  1. Looking up the file extension configuration.
  2. Retrieving the sorting and file name patterns (using extension-specific overrides if available).
  3. Resolving metadata placeholders in the patterns using ParseTemplate.
  4. Applying global string replacements from the configuration.
  5. Constructing the final target path.

The target path is set in the FIREDbRecord.TargetFilePath property. If the patterns or resolution fails, the target path remains null.

Exceptions
ObjectDisposedExceptionThrown if this instance has been disposed.

FileInfoMetadataSource IMetadataSource

Metadata source for extracting embedded file metadata using ExifTool.

ExifToolMetadataSource integrates SharpExifTool to provide access to all embedded metadata in files (EXIF, IPTC, XMP, and other formats). It supports a wide range of file types and extracts comprehensive metadata that is typically used for photo/video sorting and management.

Extracted Metadata: This source can extract metadata such as:

  • Camera make, model, and lens
  • Capture date/time (DateTimeOriginal, DateTimeDigitized, etc.)
  • Image dimensions, orientation, color space
  • GPS coordinates and location data
  • XMP comments, keywords, ratings

Async/Await Integration: SharpExifTool uses async APIs, but this implementation synchronously blocks using GetAwaiter().GetResult() to fit the synchronous interface contract.

Error Handling: If the file does not exist, ExifTool is unavailable, or metadata extraction fails, the source returns an empty or partial metadata dictionary. Exceptions are caught to prevent disrupting the collection pipeline.

Resource Management: This class implements IDisposable to clean up the SharpExifTool instance. Always dispose of this source when done to release system resources.

Registration: This source is automatically registered when a MetadataSourceRegistry is created. It is referenced with the source name "EXIFTOOL" in configuration files.

Using ExifToolMetadataSource directly:

using (var exifSource = new ExifToolMetadataSource())
{
var metadata = exifSource.ExtractMetadata("photo.jpg");
foreach (var kvp in metadata)
{
Console.WriteLine($"{kvp.Key}: {kvp.Value}");
}
}

◆ string

enum FileInfoByHandleClass string

Defines an abstraction for metadata extraction from files.

IMetadataSource establishes a contract for extracting key-value metadata from files. Implementations provide different metadata origins, such as file system properties or embedded EXIF/IPTC/XMP data. Multiple sources can be registered in MetadataSourceRegistry and used during the collection phase.

Metadata Sources: Different implementations provide different metadata origins:

  • FileInfoMetadataSource: File system properties (CreationTime, ModificationTime, AccessTime)
  • ExifToolMetadataSource: Embedded file metadata via ExifTool (EXIF, IPTC, XMP)

Error Handling: Implementations should gracefully handle missing files or extraction failures by returning an empty or partial metadata dictionary. Exceptions should be caught to prevent disrupting the collection pipeline.

Extracts all available metadata key-value pairs from the specified file.

Parameters
filePathAbsolute path to the file from which to extract metadata.
Returns
A dictionary of metadata key-value pairs. Keys are case-insensitive tag/property names, values are string representations of the metadata. Returns an empty dictionary if the file does not exist or extraction fails.

This method is called during the collection phase for each discovered file. Implementations should handle missing files and extraction errors gracefully.
