FIRE — File Information Reorganizer and Extractor

Overview

FIRE is a .NET 10 library and command-line application for automated file reorganization based on embedded metadata. It reads file system information and rich EXIF/XMP metadata (via ExifTool), stores the results in a local SQLite database, computes template-driven target paths, and finally copies, moves, or hard-links the files to their new location.

Architecture

The solution consists of three projects:

Project Type Responsibility
FIRE Class library (.NET 10) Core engine: configuration, catalog, database, localization, progress/state events
FIRE.Console Console application (.NET 10) CLI adapter for API testing and quick start (collect, generate, execute, inspect)
FIRE.Tests Test project (.NET 10) Unit tests for core library behavior

UI apps should consume the FIRE library directly (not the console process) and subscribe to catalog progress/state events.

Core Classes

Class Description
FIRE::FIRECatalog Orchestrates collection, path generation, and file operations
FIRE::FIREDatabase SQLite-backed list of FIRE::FIREDbRecord entries
FIRE::FIREConfigration Deserialized YAML configuration root
FIRE::IMetadataSource Abstraction for metadata providers
FIRE::FileInfoMetadataSource File-system metadata provider (creation/modification/access time)
FIRE::ExifToolMetadataSource ExifTool-based metadata provider (EXIF, XMP, IPTC, …)
FIRE::MetadataSourceRegistry Registry that maps source names to IMetadataSource instances
FIRE::FIRECatalogProgressEventArgs Event payload for API progress/status notifications

API Localization

API runtime messages are localized via resource files in FIRE/Resources and resolved by FIRE::Localization::ApiLocalizer.

  • Default/fallback language: English (en-US)
  • Additional cultures: de-DE, fr-FR, fil-PH

Three-Step Workflow

FIRE operates in three distinct, sequential steps:

Step 1 — collect

Scans all directories listed in FilesRootPath, extracts metadata for each file through the registered IMetadataSource implementations, and persists the results as FIRE::FIREDbRecord entries in the SQLite database.

FIRE.Console collect --config Configuration.yaml --culture de-DE

Step 2 — generate

Reads the database, evaluates the SortingPatern and FileNamePatern templates by substituting {Keyword} placeholders with resolved metadata values, and writes the computed target path back into each record.

FIRE.Console generate --config Configuration.yaml --culture de-DE

Step 3 — execute

Performs the actual file operation (Copy, Move, or Link) from the source path to the target path stored in every database record.

FIRE.Console execute --config Configuration.yaml --culture de-DE

Optional metadata inspection (inspect)

inspect reads one file, prints metadata source statistics, and writes a Markdown report for quick diagnostics and API exploration.

FIRE.Console inspect --config Configuration.yaml --culture en-US --file D:\Photos\IMG_1234.jpg

CLI Culture and Output Options

The console supports modern culture codes and localized CLI text output.

  • Recommended culture codes: en-US, de-DE, fr-FR, fil-PH
  • --culture controls both UI text language and date formatting
  • --no-wrap disables line wrapping and clips long console lines

Configuration

All behaviour is driven by a single YAML file. A minimal example:

ConfigurationVersion: 1.20
FilesRootPath:
- D:\Photos\Import
DataBasePath: D:\Photos
DataBaseFileName: FIRE.db
Action: Copy
MediaRootPath: D:\Photos\Sorted
SortingPatern: "{MediaRootPath}\\{MetaCreationTime.Year}\\{Make}\\{Model}"
FileNamePatern: "{MetaCreationTime.Year}-{MetaCreationTime.Month}-{MetaCreationTime.Day}_{Counter:D3}.JPG"
FileExtensions:
.jpg:
FileType: Image
FileClass: RegularFile
Action: Copy
AvailableKeyWords:
Make:
DataType: STRING
Source: EXIFTOOL
KeyWords:
- IFD0:Make
MetaCreationTime:
DataType: DATETIME
Source: EXIFTOOL
ValAttribute: LOWEST
KeyWords:
- ExifIFD:DateTimeOriginal
- IFD0:DateTime

Keyword Selection

When multiple EXIF tags can satisfy the same logical keyword (e.g. DateTimeOriginal and DateTime), FIRE picks one value according to ValAttribute:

ValAttribute Behaviour
LOWEST (default) Returns the chronologically or numerically smallest value
HIGHEST Returns the chronologically or numerically largest value

Supported DataType values: STRING, INT, INTEGER, DATETIME, DATE, TIME.

Metadata Sources

Source Name Description
FILEINFO File system timestamps (CreationTime, ModificationTime, AccessTime)
EXIFTOOL Full metadata extracted by ExifTool (EXIF, XMP, IPTC, QuickTime, …)

Template Placeholders

Placeholders in SortingPatern and FileNamePatern use {KeywordName} syntax. Sub-properties are separated by a dot, e.g. {MetaCreationTime.Year}. {Counter...} is only used when explicitly present in the template, and the last used values are persisted so numbering continues across restarts.

Placeholder Example output
{FileName} IMG_1234.jpg
{FileName.Noext} IMG_1234
{Make} Apple
{Model} iPhone 15 Pro
{MetaCreationTime.Year} 2026
{MetaCreationTime.Month} 07
{MetaCreationTime.Day} 04
{MediaRootPath} configured root path
{Counter:D3} sequential number per target path (e.g. 001, 002, 003)

Metadata Inspection Helpers

FIRE::FIRECatalog exposes helper APIs to inspect metadata before writing AvailableKeyWords mappings in YAML configuration.

GetAllAvailableMetadata

GetAllAvailableMetadata(string filePath) returns all discovered metadata entries for one file as tuples:

  • Source (e.g. FILEINFO, EXIFTOOL)
  • Key (metadata key/tag name)
  • Value (string representation)

Notes:

  • Returns an empty list if the file does not exist.
  • Continues with other sources if one source fails.
  • Useful for metadata discovery.
var metadata = catalog.GetAllAvailableMetadata(@"D:\Photos\IMG_1234.jpg");
foreach (var (source, key, value) in metadata)
{
Console.WriteLine($"[{source}] {key}: {value}");
}

WriteMetadataToMarkdown

WriteMetadataToMarkdown(string filePath, string? outputPath = null) writes a Markdown report containing:

  • File information (path, name, size, timestamps)
  • Metadata grouped by source (FILEINFO, EXIFTOOL)
  • Key/value tables
  • Summary statistics

Notes:

  • If outputPath is omitted, a .md file is created next to the source file.
  • The report is still generated when no metadata entries are available.
// Creates "IMG_1234.md" next to the source file
catalog.WriteMetadataToMarkdown(@"D:\Photos\IMG_1234.jpg");
// Custom report path
catalog.WriteMetadataToMarkdown(
@"D:\Photos\IMG_1234.jpg",
@"D:\Reports\IMG_1234-metadata.md");

Sidecar Files

FIRE automatically processes sidecar files (e.g. .xmp, .pp3) alongside their primary files through a four-phase workflow:

Configuration

Define sidecar extensions in the primary file's configuration:

FileExtensions:
.jpg:
SidecarFileExtensions:
- .xmp
- .pp3

Collection Phase

After adding a primary file record to the database, FIRE::FIRECatalog::ProcessSidecarFiles() searches the same directory for files matching the configured extensions and the same base name. Each found sidecar is added as a separate FIRE::FIREDbRecord with:

  • FIRE::FileClassification::SidecarFile classification
  • Cloned metadata from the primary file

Path Generation Phase

FIRE::FIRECatalog::GenerateTargetPaths() treats sidecar records specially:

  • Locates the primary file by matching source directory and base name
  • Inherits the primary file's target directory and base filename
  • Preserves the sidecar's own extension

Example:

  • Primary: D:\Import\IMG_1234.jpg → D:\Sorted\2026\07\2026-07-04_001.jpg
  • Sidecar: D:\Import\IMG_1234.xmp → D:\Sorted\2026\07\2026-07-04_001.xmp

Execution Phase

FIRE::FIRECatalog::ExecuteFileOperations() processes sidecar files using the global action setting (Copy/Move/Link) from the configuration root, since sidecar extensions typically have no dedicated extension configuration.
